Design a module called TopModule. This module is an SRAM controller: it manages access to embedded
SRAM with transparent scrambling and integrity protection, handles keys and nonces from OTP, and gates
access based on lifecycle state.

Overview

TopModule is an advanced SRAM controller supporting multiple instances of embedded SRAM with built-in
security features. It provides transparent encryption/decryption of data using PRINCE cipher with
supplied keys and nonces from the OTP macro. The module verifies data integrity using error-correcting
codes (ECC) and reports errors. It includes a separate register-based control interface for configuration
and a memory-mapped interface for SRAM read/write access. The module operates with dual clocks (main
and OTP), has lifecycle-gating for initialization and debug access, and supports RACL policies for
register protection. It tracks initialization state via a counter and permits execution-in-place (XIP)
when initialized and lifecycle permits.

Parameters

  Parameter   Meaning   Constraint  
 ----------- --------- ------------ 
  MemSizeRam   Total SRAM size in bytes.   Typical: 0x1000 (4 KB).  
  InstSize   Instantiated SRAM size per instance.   Typical: equal to MemSizeRam; must be multiple of word width.  
  NumRamInst   Number of SRAM instances.   Typical: 1; can be > 1 for multi-bank designs.  
  AlertAsyncOn   Alert asynchronous enable mask.   Width NumAlerts.  
  AlertSkewCycles   Clock cycles of alert timing skew.   Typical: 1.  
  InstrExec   Enable instruction execution from SRAM.   0 = data only, 1 = permit execution.  
  NumPrinceRoundsHalf   Half-rounds in PRINCE cipher.   Typical: 3 (6 full rounds).  
  Outstanding   Max outstanding memory transactions.   Typical: 2.  
  EccCorrection   Enable single-error correction.   0 = detection only, 1 = correction.  
  EnableRacl   Enable role-based access control.   0 = disabled, 1 = enabled.  
  RaclErrorRsp   RACL error response mode.   Defaults to EnableRacl.  
  RaclPolicySelVecRegs   RACL policies for registers.   Array per register.  
  RaclPolicySelNumRangesRam   Number of RACL ranges for RAM.   Typical: 1 (no protection) or > 1 (partitioned).  
  RndCnstSramKey   Constant SRAM key for PRINCE.   128-bit constant from design.  
  RndCnstSramNonce   Constant SRAM nonce.   64-bit constant from design.  
  RndCnstLfsrSeed   Constant LFSR seed.   For initialization counter.  
  RndCnstLfsrPerm   Constant LFSR permutation.   For random address scrambling during init.

Behavioral requirements

- Dual TileLink interfaces. TopModule exposes two independent TileLink register buses: one for SRAM memory access (ram_tl_i/ram_tl_o) and one for control registers (regs_tl_i/regs_tl_o). They operate in the main clock domain with independent ready/valid handshaking.

- Transparent encryption/decryption. All SRAM data is encrypted at write time and decrypted at read time using PRINCE cipher in counter (CTR) mode. The key comes from the OTP via a request/response interface; the nonce is derived from the address and design constants.

- ECC/integrity protection. Each SRAM word includes an ECC code computed during write. On read, the ECC is checked and reported in sram_rerror_o. If EccCorrection = 1, single-bit errors are silently corrected; if 0, all errors are reported and data is likely corrupted.

- Key management. The module requests keys and nonces from the OTP via sram_otp_key_o and receives them on sram_otp_key_i. The request includes key version and purpose; responses include the key and validity. Keys are stored internally and used for encryption.

- Lifecycle gating. The lc_escalate_en_i, lc_hw_debug_en_i, and otp_en_sram_ifetch_i inputs control whether certain operations (e.g., initialization, instruction fetch) are permitted. Denied access results in error responses.

- Initialization flow. On reset or lifecycle change, the SRAM must be initialized to remove data remnants. This is performed by writing a random or zero pattern to all addresses using a pseudo-random address sequence. An initialization counter (in register space) tracks progress and is disabled once complete.

- Multiple instances and RACL. When NumRamInst > 1, the module can address multiple separate SRAM banks. The cfg_i array configures each instance independently. The racl_policy_sel_ranges_ram_i input allows partitioning RAM address ranges with different access policies.

- Error reporting. Read errors (detected via ECC) are logged in sram_rerror_o with syndrome and valid flag. Alerts are raised for critical errors or policy violations. Register access violations are logged in racl_error_o.

Example

Write and read a 32-bit word with encryption:
- Write request: address 0x100, data 0xDEADBEEF via ram_tl_i.
- Controller requests key from OTP, receives 128-bit key + nonce.
- Data is XOR'd with PRINCE-CTR keystream derived from address 0x100.
- ECC code is computed on encrypted data.
- Write completes; SRAM stores encrypted data + ECC.
- Read request: address 0x100.
- Encrypted data + ECC is read from SRAM.
- ECC is verified (no error, or corrected if single-bit error).
- Data is decrypted using same PRINCE-CTR keystream.
- Result 0xDEADBEEF is returned to reader.
